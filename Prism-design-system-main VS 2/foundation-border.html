Border

Use borders to define boundaries, communicate interaction states, and improve scan clarity across endpoint workflows.

Border principles

1. Borders communicate boundaries

Borders should separate meaningfully different content regions. Do not use borders where whitespace already creates clear grouping.

Good use cases
Table row separators Filter sidebar edges Drawer boundaries Destructive confirmation zones Widget containers Split troubleshooting panes

2. Pair width and color tokens

Prism borders are always created by combining a width token and a semantic color token. A border without semantic meaning becomes visual noise.

✓ Good
border.width.default + color.border.subtle
border.width.selected + color.border.selected
border.width.focused + color.border.focused
✗ Bad
border: 1px solid #ccc
border: 2px solid rgba(0,0,0,0.1)
border-bottom: 1px solid gray

Border width tokens

Width tokens define the thickness of borders. These come from the UEMS Global Tokens collection and are consistent across all themes.

Token Value Preview Usage
border-width-0 0px
No border — reset/remove
border-width-1 1px
Default — table rows, cards, dividers
border-width-2 2px
Emphasis — selected, focused, active states

Anti-patterns

✗ Don't
  • Hardcode hex colors for borders — always use tokens
  • Use borders where whitespace suffices
  • Mix border-width values without tokens
  • Use border: 1px solid #ccc — this breaks in dark mode
  • Apply accent borders for non-interactive elements
✓ Do
  • Pair width + color tokens for every border
  • Use semantic variants (success, error, warning) for status
  • Test borders in both light and dark themes
  • Use Border-Accent-Focus for keyboard focus rings
  • Choose the right hierarchy (Primary → Quaternary)

Principle

"A border is a promise of separation. Break that promise, and you break trust."

Every border in Prism exists because a boundary was needed — not because a pixel was empty.
