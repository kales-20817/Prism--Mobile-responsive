Elevation

Use elevation to define surface hierarchy, guide attention, and communicate interaction across Endpoint Central workflows.

In Prism, elevation represents layered surfaces that form the foundation of the UI. These layers create the structure where content, actions, and interactions live.

A strong elevation system improves clarity, focus, and usability across dense enterprise interfaces.

Elevation principles

1. Use elevation to show hierarchy

Elevation defines which surface is base, interactive, floating, or blocking.

Rule: Higher elevation = higher priority.

  • page background → lowest
  • cards → medium
  • dropdown → high
  • modal → highest

2. Use elevation sparingly

Too many elevation levels create visual noise.

  • limit to defined elevation steps
  • avoid stacking multiple shadows
  • do not combine heavy borders + shadows
  • use spacing before adding elevation
Whitespace first → elevation second.

3. Elevation must be consistent

The same interaction should always use the same elevation level.

  • all dropdowns → same elevation
  • all modals → same elevation
  • all tooltips → same elevation

Inconsistent elevation breaks user expectation.

Elevation layers

Prism defines a structured elevation scale.

Shadow system

Elevation is visually represented using shadows and surface contrast.

  • use tokenized shadow values only
  • avoid custom shadows
  • combine shadow + background contrast
  • ensure shadows work in dark mode
  • avoid heavy blur shadows in dense tables

Endpoint-specific elevation patterns

Device inventory

  • base table → default surface
  • selected row actions → raised surface
  • inline editors → overlay

Inspector panel

  • main content → default
  • inspector rail → raised
  • nested sections → subtle elevation only

Patch deployment

  • step container → raised
  • dropdown selection → overlay
  • confirmation modal → modal elevation

Security workflows

  • dashboard widgets → raised
  • threat detail popover → overlay
  • isolation confirmation → modal

AI insights

  • suggestion cards → raised
  • explanation popovers → overlay
  • action preview → overlay or modal based on risk

Elevation + interaction

Hover

  • slight elevation increase
  • subtle shadow only

Active / pressed

  • reduce elevation
  • simulate "pressed surface"

Focus

  • do not rely on elevation
  • use border + focus ring instead

Elevation + grid and spacing

Elevation should not replace layout structure.

  • use grid for layout
  • use spacing for grouping
  • use elevation only for layering
DON'T

Using shadows to separate sections.

DO

Spacing + grid define layout; elevation defines interaction depth.

Accessibility

Elevation must remain perceivable for all users.

  • never rely on shadow alone
  • pair elevation with border, contrast, background shift
  • ensure visibility in dark mode
  • maintain clear focus states
  • support reduced motion environments

Anti-patterns

Avoid these common enterprise mistakes:

  • too many shadow levels
  • heavy drop shadows on tables
  • inconsistent modal elevation
  • nested cards with identical elevation
  • using elevation instead of spacing
  • invisible shadows in dark mode
  • mixing blur styles across modules

Principle

In Prism, elevation creates spatial clarity.

The right elevation system helps Endpoint Central users understand structure, focus on what matters, and confidently interact across complex layered workflows.
