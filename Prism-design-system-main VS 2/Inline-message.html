Inline Message

A semantic banner that communicates status information within the page flow — below a form, inside a card, or at the top of a panel. It supports five semantic types, two visual weights, optional title, description, action link, and dismiss control, with full RTL support.

Content GuidelinesRecommended

Do
  • Keep the title concise (3–6 words: "Action Required", "Upload Failed").
  • Use description to provide context and explain next steps.
  • Use Show Action to link directly to the resolution ("Retry", "View error details").
  • Match type to the actual semantic meaning — Error for failures, Success for completions.
Don't
  • Write a full sentence as the title — use description for longer explanatory text.
  • Default to Intense for all alerts — reserve it for high-priority messages.
  • Use Warning for errors or Info for success — mismatched semantics confuse screen readers.
  • Put unrelated page-level actions in the alert action slot.

When and Where to Use

Inline alert messages communicate status or feedback directly within the page content flow.

When to use

  • To communicate a status or feedback message directly within the page content flow.
  • Use Info for neutral guidance, tips, or informational notices.
  • Use Success to confirm a completed action within a page region.
  • Use Warning when the user should proceed with caution.
  • Use Error to report a validation failure or blocked action.
  • Use Subtle (default) for most in-page messages.
  • Use Intense when the message demands immediate visual priority.

When NOT to use

  • Global/system-wide alerts — use a Toast Message or page-level banner.
  • Field-level validation messages — use the form field's built-in helper/error text.
  • Transient confirmations that auto-dismiss — use Toast Message.
  • Blocking modal errors — use a Dialog / Modal.
  • Progress or loading states — use Progress Bar or a loading indicator.

Alignment & Placement

Consistent placement ensures inline alerts are predictable and scannable.

  • Below a form: Position the alert directly below the form field group, spanning the full width of the form container.
  • Inside a card: Place at the top of the card content area, above the primary content.
  • Panel top: Position at the top of a panel or section, below the header but above content.
  • Full width: The component stretches to 100% of its parent container width.

Complete Do's and Don'ts

Guidelines for effective inline alert message usage.

Do
  • Use Subtle for most inline notifications: It fits naturally within page content.
  • Match type to semantic meaning: Error for failures, Success for completions.
  • Keep the title concise: 3–6 words like "Action Required" or "Upload Failed".
  • Use description for context: Explain what happened and next steps.
  • Use Show Action: Link directly to the resolution ("Retry", "View error details").
  • Use Show Dismiss: For temporary, non-critical messages the user can safely ignore.
  • Use RTL=True: For right-to-left pages; set dir="rtl" on the container.
  • Stack multiple alerts: When multiple distinct issues need reporting.
Don't
  • Don't default to Intense: Reserve it for high-priority messages that need to stand out.
  • Don't mismatch semantics: Using Warning for errors or Info for success confuses screen readers.
  • Don't write full sentences as titles: Use description for longer explanatory text.
  • Don't show only a title: When the situation requires explanation, add a description.
  • Don't put unrelated actions: In the alert action slot.
  • Don't hide dismiss on non-critical: Allow users to close temporary messages.
  • Don't manually mirror layout: Use the RTL variant instead of CSS overrides.
  • Don't repeat the same message: Consolidate consecutive banners of the same type.

RTL Support

  • In RTL mode, the child order is reversed: Dismiss Icon → Content → Status Icon (right-to-left reading order).
  • Title and description text alignment changes to text-align: right.
  • Set dir="rtl" on the component or its page container in code.
  • Arabic placeholder text is used in the Figma component for RTL variants.

Accessibility

Guidelines for ensuring inline alert messages are accessible across assistive technologies.

Keyboard Interaction

KeyAction
TabMoves focus to the dismiss button (if visible) and action link (if visible).
Enter / SpaceActivates the focused dismiss button or action link.
Shift+TabMoves focus backwards through interactive elements within the alert.

If Show Dismiss is true, the dismiss button must be reachable via Tab and activatable with Enter / Space. The action link must also be keyboard-accessible.

ARIA Attributes & Screen Reader Guidance

  • ARIA role: Render as role="alert" for Error/Warning (live region announced immediately) or role="status" for Info/Success/Neutral (polite announcement).
  • Live region: aria-live="assertive" for Error and Warning; aria-live="polite" for Info, Success, and Neutral.
  • Status icon: The status icon is decorative — mark it aria-hidden="true". Semantic meaning is conveyed by the title and role.
  • Dismiss button: Must have a descriptive accessible name: aria-label="Dismiss alert" or equivalent.
  • Action link: The action label "Action" alone may be ambiguous — use aria-label with context if needed.
  • Dynamic injection: When an alert appears dynamically (e.g., after form submission), inject it into a live region container already in the DOM so screen readers detect the change.

Focus State Design

  • The dismiss button and action link receive visible focus rings when navigated via keyboard.
  • Focus order within the alert: Action Link → Dismiss Button (natural DOM order).
  • After dismissal, focus should move to the next logical element in the page flow.
