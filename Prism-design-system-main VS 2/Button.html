Button

The Button is the primary interactive element for triggering actions. It supports multiple visual variants for semantic meaning, four sizes for layout flexibility, and five interaction states.

Content GuidelinesRecommended

Do
  • Use Primary for the main call-to-action on a page.
  • Use Destructive for irreversible actions.
  • Keep labels concise (1-3 words).
Don't
  • Use more than one Primary button per section.
  • Rely solely on color communicating state or meaning.

When and Where to Use

A button triggers an action or an event, such as submitting a form, opening a dialog, canceling an action, or performing a delete operation.

When to use

  • To trigger a primary or secondary action on the page (e.g., Save, Submit, Delete).
  • To submit a form or confirm a modal dialog.
  • To step through a multi-step flow (e.g., Next, Previous).
  • To perform a state change in the application interface.

When NOT to use

  • For navigation to another page. Use an <a> element styled as a Link instead.
  • When the action is just to toggle visibility (consider a Toggle or Accordion).
  • If the primary focus is choosing one of many options (use Radio buttons or a Dropdown).

Alignment & Placement

Consistent placement makes it easier for users to locate actions without scanning everything on the screen.

  • Forms: Place submit buttons on the bottom-left aligned with the input fields, or right-aligned in specialized filter scenarios.
  • Modals/Dialogs: Place primary and secondary actions at the bottom-right of the dialog window. The primary button should ALWAYS be on the far right.
  • Data Tables: Global actions (e.g., "Add User") should be placed on the top-right above the table controls.
  • Page Headers: Primary page-level actions belong in the top-right corner of the page header banner.

Complete Do's and Don'ts

Visual and interaction guidelines to ensure buttons are effective, scalable, and accessible.

Do
  • Limit Primary buttons: Keep to one Primary button per screen or modal view to provide clear focus.
  • Use alternative variants: Rely on Outline or Secondary variants for alternative actions.
  • Logical grouping: Maintain locally ordered button groups. The primary action should always take the visual lead.
  • Provide feedback: Include a loading spinner state instead of abruptly disabling the button when an action is processing.
  • Accessibility tags: Set aria-label for Icon-only buttons so screen readers can process them correctly.
Don't
  • Avoid long labels: Don't use excessively long labels. Keep the text between 1-3 highly descriptive words.
  • No text wrapping: Don't wrap text inside a button. If the label is too long, the button form-factor shouldn't be used.
  • Don't rely solely on color: Don't communicate intent (e.g. Danger) entirely through color. Use an icon or explicitly descriptive text.
  • Misusing destructive style: Don't place destructive functionality inside a standard secondary regular button.

Variants

Buttons convey their hierarchy and semantic meaning through distinct visual styles. A page should typically only feature one primary action.

Variant Usage Example
Primary The primary call to action. Use only one per view.
Secondary Alternative actions. Often placed next to a primary button.
Outline Secondary actions that need slightly more emphasis than default.
Destructive Dangerous, irreversible actions like Delete or Remove.
Tertiary Low emphasis actions, often used in less prominent areas.
Success Confirmative actions that represent a successful state or progress.
Warning Actions requiring caution, but not necessarily destructive.

Implementation Idea: Ember Hook Form

A common use case in modern Ember applications involves linking the button directly to form changeset or submission states.

<!-- MyApp/templates/components/submit-form.hbs -->
<UemsButton
  @variant="primary"
  @size="l"
  @isLoading={{this.isSubmitting}}
  type="submit"
>
  Save Changes
</UemsButton>

RTL Support

  • When RTL = True, layout direction is mirrored — prefix icon moves to trailing, suffix to leading.
  • Text direction set to auto. All spacing and padding values are mirrored.
  • Icon positions swap accordingly to maintain visual hierarchy.

Accessibility

Guidelines for ensuring all button interactions are accessible across assistive technologies.

Keyboard Interaction

KeyAction

ARIA Attributes & Screen Reader Guidance

Ensure button intent is explicitly communicated to all assistive technologies.

  • Use the native <button> element to inherit default accessibility behaviors. Avoid using <div> or <a> tags.
  • Icon-Only Buttons: Must explicitly define an aria-label describing the action to ensure screen reader support.
  • Disabled State: Use aria-disabled="true" instead of the native HTML disabled attribute if you need the button to remain focusable so tooltips can explain why it is disabled.
  • Loading State: Add aria-busy="true" and consider an aria-live region announcement.
  • State Toggling: For toggle buttons (like Pin vs Unpin), use aria-pressed="true" or aria-expanded="true" depending on the behavior.
  • Don't rely solely on color to communicate the button variant status — always pair with a clear label or descriptive icon.

Focus State Design

  • Focus ring: 2px solid using the variant's accent color (Primary: #006AFF).
  • Outer border radius: 12px, inner padding: 4px from focus ring.
  • Inner container retains the default state background.

WCAG 2.1 Level AA Requires: A contrast ratio of at least 4.5:1 for normal text (like most button labels) and 3.0:1 for large text and UI components (like icons and focus rings). All UEMS core buttons pass these standards.
