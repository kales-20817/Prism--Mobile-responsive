Page Header

A full-width layout component that anchors every page. It provides a consistent location for the breadcrumb trail, page title, contextual actions, metadata summary, and tab-based navigation. Four Structure variants progressively compose these zones to match page complexity.

When and Where to Use

Choose the right Page Header structure based on page complexity and content requirements.

When to use

  • As the top-level header for every page in the application.
  • Use Default for simple pages with no contextual metadata or tab navigation.
  • Use With Summary when the page has key metadata that users frequently reference.
  • Use With Tabs when a single page contains multiple distinct content sections.
  • Use Full for complex record or detail pages requiring both metadata and tabs.

When NOT to use

  • Inside a card, modal, or panel — use Section Header or a custom heading instead.
  • For secondary/nested pages where breadcrumbs would exceed 3 levels.
  • As a section header within a page — use Section Header.

Complete Do's and Don'ts

Guidelines for effective Page Header usage.

Do
  • Match complexity: Choose the Structure that matches page complexity — start with Default.
  • Short title: Keep the title to one short, descriptive noun phrase (2–4 words).
  • Chevron purpose: Use showChevron=true only when the title triggers a dropdown/context menu.
  • Status badge: Use showBadge=true to communicate the record's current status.
  • Back icon: Use showBackIcon=true on modal-like full-page detail views with a clear parent.
  • Short labels: Keep summary labels short (1–2 words: "Status", "Owner", "Due").
  • Limit metadata: Limit Summary Section to the 4 most relevant metadata fields.
  • Primary action: Use the primaryAction for the single most important page-level action.
  • Overflow menu: Put destructive or rare actions behind the overflow menu.
  • Dynamic counts: Set meaningful tab count badges that update dynamically.
Don't
  • Don't over-use Full: Default to Full on every page regardless of content needs.
  • No sentences: Use a full sentence or question as the page title.
  • No decorative chevrons: Show the chevron as decoration without a real interaction.
  • No stacked badges: Stack multiple badges in the title area.
  • No dual navigation: Show both breadcrumbs and a back icon simultaneously.
  • No long labels: Write descriptive phrases as metadata labels.
  • No overflow: Add more than 4 metadata items — the layout overflows.
  • Limit buttons: Put more than 2 labelled buttons in the actions area.
  • No destructive top-level: Surface destructive actions as top-level buttons.
  • No empty counts: Show 0 in every tab count badge.

Summary Section (Metadata Row)

The Summary Section contains 4 fixed metadata item slots with a 24 px gap between them.

SlotDefault LabelDefault Value TypeNotes
1StatusStatus Indicator componentDot + label (e.g. "Active"). Success/Error/Warning states supported.
2CreatedPlain textDate string (e.g. "Mar 15, 2026").
3OwnerPlain textUser name (e.g. "John Doe").
4PriorityBadge componente.g. "Critical". Instance-swappable.

Slots 1 and 4 use embedded component instances (Status Indicator and Badge) rather than plain text values. Replace with appropriate component instances when customising.

Tab Bar

The embedded Tab Bar instance exposes 5 tab slots.

  • Each tab contains an icon (20 x 20 px, instance-swappable), a label text (default "Tab Menu"), and a count badge pill (default "34" on active, "66" on others).
  • Tab state: The first tab is Active; all others are Default. The active tab uses Text/Text-Accent-Primary and Border/Border-Accent underline.

Child Layer Structure

Page Header (VERTICAL, paddingTop/Bottom: 12 px, gap: 12 px)
└── Content (VERTICAL, FILL width, gap: 12 px)
    ├── Breadcrumbs row (HORIZONTAL, paddingL/R: 20 px)
    ├── page title content (HORIZONTAL, FILL, paddingL/R: 20 px, gap: 8 px)
    │   ├── Title Row (gap: 4 px)
    │   └── Actions (HORIZONTAL, HUG, gap: 8 px)
    ├── Summary Section [With Summary + Full]
    ├── Divider [1 px, Border-Tertiary — With Summary]
    └── Tab bar [With Tabs + Full]

RTL Support

  • RTL is handled at sub-component level (Breadcrumb, Tab Bar, Buttons each have RTL variant properties).
  • Set dir="rtl" on the page <html> or <header> element.
  • No additional variant axes exist on the Page Header component set itself for RTL.

Accessibility

Guidelines for ensuring the Page Header component is accessible across assistive technologies.

Landmark & Semantic Structure

  • Landmark: Wrap the Page Header in <header> for page-level landmark semantics.
  • Heading level: The page title should render as <h1>. There must be exactly one <h1> per page.
  • Skip link: Provide a "Skip to main content" link that jumps past the Page Header for keyboard and screen reader users.

ARIA Attributes & Screen Reader Guidance

ElementRequirement
BreadcrumbWrap in <nav aria-label="Breadcrumb">. Mark the current page item with aria-current="page".
ActionsEach button must have a visible label or aria-label. The overflow button needs aria-label="More actions" and aria-haspopup="true".
ChevronIf the title chevron opens a menu, use aria-haspopup="menu" and aria-expanded on the trigger.
Star / FavouriteUse aria-label="Add to favourites" / aria-pressed (toggle button).
TabsUse role="tablist", role="tab", aria-selected, and aria-controls. Active tab must have aria-selected="true".
Count badgesAssociate count text with tabs via aria-label, e.g. aria-label="Tab Menu, 34 items".
Summary labelsUse <dl> / <dt> / <dd> structure for metadata key-value pairs.

Keyboard Interaction & Focus Order

KeyAction
TabMoves focus through interactive elements in reading order: breadcrumb links → title chevron → star toggle → action buttons → tab items.
Enter / SpaceActivates the focused interactive element (button, link, tab, toggle).
Arrow Left / RightNavigates between tabs when the tablist is focused.

Tab order should follow visual reading order: breadcrumb → title actions (chevron, star) → buttons → tabs.

Color Contrast MinimumsWCAG AA

All text and interactive elements must meet WCAG 2.1 AA contrast requirements.

  • Text/Text-Quaternary (breadcrumb ancestors) must meet WCAG AA 4.5:1 on the page background.
  • Primary button: white text on blue gradient meets AAA contrast.
  • Active tab label and underline use accent colour tokens with sufficient contrast.
  • Badge text colours (success green, error red) are validated against their respective background tints.

RTL Support

  • RTL is handled at sub-component level — Breadcrumb, Tab Bar, and Buttons each have RTL variant properties.
  • Set dir="rtl" on the page <html> or <header> element.
