Radius

Use radius tokens to define corner roundness, improve containment clarity, and create a trustworthy enterprise visual language across endpoint workflows.

Radius principles

1. Radius communicates containment

The larger the surface, the more it benefits from clearer corner softness. This helps users visually separate chips from cards, cards from drawers, drawers from modals, and modals from full-page containers.

2. Radius must scale with component role

Prism radius must map to component intent instead of arbitrary styling. Use semantic token names that describe purpose, not pixel values.

✓ Good
radius-xs → status dots, tiny badges
radius-default → buttons, inputs
radius-m → cards, dropdowns
radius-l → modals, large tables
radius-pill → avatars, pill buttons
✗ Bad
border-radius: 4px
border-radius: 8px
border-radius: 50%
border-radius: 0.5rem
border-radius: 10px

Token alias chain

Radius tokens follow the same three-tier architecture as color tokens. Semantic tokens reference primitives, and primitives resolve to raw pixel values.

Component
border-radius: var(--radius-s)
Semantic
radius-s → radius-8
Primitive
radius-8 → 8px

Component mapping

Each component type maps to a specific radius token. This ensures consistent visual hierarchy regardless of where the component appears.

Component Token Value Preview
Table cells, inline tags radius-none 0px
Checkboxes, status dots radius-xs 4px
Buttons, inputs, selects radius-default 6px
Cards, panels, tooltips radius-s 8px
Dialogs, drawers, popovers radius-m 12px
Hero cards, feature panels radius-l 16px
Page sections, onboarding radius-xl 20px
Badges, chips, avatars, pills radius-pill 9999px

Anti-patterns

✗ Don't
  • Hardcode pixel values — always use radius tokens
  • Use border-radius: 50% for non-circular elements
  • Mix radius values within the same component group
  • Apply large radii to small elements (e.g., radius-xl on a checkbox)
  • Use sharp corners on elevated surfaces (modals, drawers)
✓ Do
  • Map each component to the appropriate semantic token
  • Scale radius with component size (small → small, large → large)
  • Use radius-pill only for badges, chips, and pill buttons
  • Keep nested elements' radii proportional to their parent
  • Test radius at all breakpoints for consistent feel

Principle

"Radius is a trust signal. Sharp corners feel technical. Rounded corners feel human. The right mix makes enterprise software feel both powerful and approachable."

Every radius in Prism exists to communicate the role and weight of its surface — not for decoration.
