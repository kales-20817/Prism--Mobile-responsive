Color

Use color to create meaning, reinforce trust, and accelerate operational decisions across endpoint workflows.

In Prism, color is not decorative. It communicates system health, risk severity, action priority, device state, policy compliance, and workflow confidence.

A strong color system helps admins scan faster, reduce mistakes, and act safely under pressure.

Color anatomy

Prism color foundation follows three layers.

Saturated colors

Used for meaning-rich UI signals.

Critical threat alerts
Successful remediation
Warning states
Brand CTAs
Active navigation
AI recommendation highlights

Neutral colors

Used for structural UI: backgrounds, surfaces, borders, tables, dividers, secondary text, disabled states, and empty states. Neutrals must support both dense light and dark environments.

Alpha colors

Transparent overlays used for adaptive surfaces: hover overlays, selected rows, focus rings, drawer scrims, severity heatmap overlays, and live telemetry pulse layers. Alpha colors must adapt safely across elevations and themes.

Token-first color system

Never use raw HEX values in Prism products. All colors must be applied through semantic tokens.

Token structure

color.{property}.{role}.{emphasis}.{state}

Collection: UEMS Theme Tokens · Supports light & dark mode switching

Semantic color roles

Every status color in Prism maps to an operational meaning in endpoint management.

Success

Healthy devices, completed patches, passed compliance, successful deployments

Error

Failed deployments, critical threats, validation failures, device isolation

Warning

Pending patches, expiring policies, drift detected, attention needed

Alert

High-severity threats, urgent escalations, active incidents

Accent

Brand actions, primary CTAs, active navigation, selected items

Acknowledge

Reviewed items, noted states, acknowledged alerts

Usage guidelines

Do
  • Always use semantic tokens, never raw hex values
  • Test all states in both light and dark themes
  • Use status colors consistently across modules
  • Ensure 4.5:1 contrast for text on backgrounds
  • Reserve red exclusively for errors and destructive actions
  • Use alpha colors for adaptive hover/selection surfaces
Don't
  • Don't hardcode hex values in component styles
  • Don't use color as the only way to convey meaning
  • Don't mix success green with brand green
  • Don't use saturated colors for large background areas
  • Don't create custom severity colors outside the system
  • Don't ignore dark mode contrast requirements

Color anti-patterns

Avoid these common enterprise color failures:

  • Using red for non-destructive actions
  • Inconsistent severity colors across modules
  • Low-contrast text on colored backgrounds
  • Saturated backgrounds that compete with content
  • Hardcoded colors that break in dark mode
  • Multiple greens for different success meanings
  • Color-only status indicators without labels or icons
  • Custom hex values outside the token system

Accessibility and color

Color directly impacts accessibility. Ensure:

  • All text meets WCAG 2.1 AA contrast (4.5:1 normal, 3:1 large text)
  • Color is never the sole indicator of state — pair with icons, labels, or patterns
  • Focus rings are visible in both themes
  • Status indicators work for color-blind users (red-green deuteranopia)
  • Dark mode preserves semantic meaning, not just inverts colors
  • Hover and active states are distinguishable without color vision

In endpoint management, a misread severity color can be an operational risk. Color must be unambiguous.

Principle

In Prism, color is operational language. It helps admins recognize severity instantly, trust the interface, and act confidently across dense endpoint workflows.
