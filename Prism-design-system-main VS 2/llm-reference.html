LLM Reference

How to expose the Prism design system to LLMs so they generate spec-compliant, token-aware UI code every time.

LLMs write code that looks correct but doesn't follow design systems. They invent colors, skip tokens, hardcode spacing, and ignore component APIs. This page documents the three-layer approach that fixes this — getting AI-generated UI to be Prism-compliant by default.

TL;DR
  • LLMs don't know your design system unless you tell them explicitly.
  • Three-layer solution: spec files for components, semantic tokens for design decisions, audit scripts to verify compliance.
  • Feed spec files + token files into every LLM prompt — or wire them into your IDE context.
  • Pair with audit scripts to catch regressions before they ship.

The problem with LLMs and design systems

LLMs are trained on vast amounts of generic code. That makes them good at writing React, HTML, and CSS — but not at following your specific system.

When you ask an LLM to "build a settings panel", it will:

What you expect What you get
--prism-text-muted for secondary text Hardcoded #9ca3af or gray-400
<Button variant="primary"> Custom <button style="background:blue">
Spacing using --spacing-3 Inline padding: 12px
Accessible label + input pairing Unlabelled input with placeholder only
Dark mode via CSS variables No dark mode or duplicate class overrides

The root cause: LLMs don't fail because they're bad at coding. They fail because they don't know what your system looks like. Without context, they fall back to generic patterns from their training data.

The three-layer solution

Structured context at three levels makes LLMs generate Prism-compliant code reliably.

Layer 01
Spec files
Markdown files that describe each component — its API, variants, anatomy, usage rules, and do/don't guidance. LLMs consume these as system prompts.
Layer 02
Token layer
A structured map of semantic design tokens — colors, spacing, radius, elevation — tied to their CSS variable names and intent. Prevents hardcoded values.
Layer 03
Audit scripts
Static analysis tools that scan generated code for compliance violations: hardcoded hex values, unknown components, missing token usage.

Layer 1 — Spec files

Every Prism component has a .md spec file that documents it for both humans and LLMs. These files follow a consistent structure:

# component-name.md structure ## Overview Short description of what the component is and its primary use case. ## Anatomy Named breakdown of each part — root, trigger, content, arrow, etc. ## Variants / Props Every supported variant with accepted values and defaults. ## Token mapping Which CSS variables control each visual property. ## Usage rules When to use this component, when not to, and accessibility requirements. ## Do / Don't Concrete examples of correct vs. incorrect usage.

Spec files live alongside component source and are checked in to version control. When a component changes, the spec changes too. This keeps LLM context in sync with the actual system.

Layer 2 — Token layer

Semantic tokens are the bridge between design intent and code. A token file tells the LLM: "use --prism-text-muted for secondary text" instead of a raw hex value.

/* prism-tokens.css — excerpt */ :root { /* Text */ --prism-text: rgba(255, 255, 255, 0.92); --prism-text-muted: rgba(255, 255, 255, 0.48); /* Surfaces */ --prism-surface: rgba(255, 255, 255, 0.05); --prism-surface-hover: rgba(255, 255, 255, 0.09); /* Brand */ --prism-violet: #2563eb; --prism-cyan: #06b6d4; /* Spacing */ --spacing-1: 4px; --spacing-2: 8px; --spacing-3: 12px; --spacing-4: 16px; --spacing-6: 24px; --spacing-8: 32px; }

Feed this token file into every LLM session as part of your system prompt. The model learns to reach for tokens first, not raw values.

Layer 3 — Audit scripts

Audit scripts are the safety net. They scan generated or committed code for violations and surface them before they reach production.

# prism-audit.sh — example checks # 1. Detect hardcoded hex colors not matching Prism palette grep -rn '#[0-9a-fA-F]\{3,6\}' src/ | grep -v 'prism-tokens' # 2. Find inline style padding/margin not using tokens grep -rn 'style=".*padding:.*[0-9]px' src/ # 3. Check for components used outside Prism component set grep -rn '<button ' src/ | grep -v 'Button'

Run these in CI on every pull request. Violations fail the build. This creates a feedback loop that trains both LLMs and engineers to stay within the system.

Architecture

Prism uses a three-layer token architecture. Each layer depends on the one below it — changes at the upstream level cascade automatically.

Layer 1 — Upstream / Primitive prism-tokens.css

Raw value definitions — the full palette, base spacing scale, radius scale, and elevation values. These are never referenced directly in components.

--color-blue-600: #2563eb; --space-4: 16px; --radius-md: 8px;
Layer 2 — Semantic / Project project-tokens.css

Intent-mapped aliases. These reference Layer 1 values and give them semantic meaning — surface, text, border — that components consume. Light/dark themes are defined here via [data-theme] overrides.

--prism-violet: var(--color-blue-600); --prism-text: rgba(255, 255, 255, 0.92); --prism-surface: rgba(255, 255, 255, 0.05); --prism-border: rgba(255, 255, 255, 0.07); [data-theme="light"] { --prism-text: #1a1a2e; --prism-surface: rgba(37, 99, 235, 0.028); }
Layer 3 — Component component.css

Component-scoped aliases that map to Layer 2 semantic tokens. Each component only knows about its own token namespace. This makes theming and overriding predictable — you never touch raw values.

/* Tooltip component tokens */ .tt-root { --tt-bg: var(--prism-surface); --tt-text: var(--prism-text); --tt-border: var(--prism-border); --tt-radius: var(--radius-md); }

Token resolution order

When a component renders, CSS resolves tokens in this order — from the most specific override back to the root primitive.

/* Resolution chain for a Tooltip background in light mode */ .tt-body { background: var(--tt-bg); } ↓ resolves to --tt-bg: var(--prism-surface) ↓ resolves to (light mode override) --prism-surface: rgba(37, 99, 235, 0.028)

What to feed the LLM — and in what order

Context order matters. Feed layers from most-general to most-specific so the model builds understanding before it sees component details.

  1. 1
    Layer 2 tokens — prism-tokens.css
    The full semantic token map. This is the most important file — it defines the complete vocabulary the LLM should use for every color, spacing, and surface decision.
  2. 2
    Relevant foundation spec(s)
    If the task involves layout: include spacing.md and grid.md. If it involves color decisions: include color.md. Don't include all foundation files — excess context dilutes focus.
  3. 3
    Relevant component spec(s) from specs/
    The specific component spec files needed for the task. Include only what's needed — if building a form: input.md, button.md, label.md.
  4. 4
    Your task prompt with explicit constraints
    Write the task after the context. Start with: "Using only Prism tokens and the components defined in the spec files above, build…"

The setup — six steps

Configure your LLM to use Prism by default. This works with Claude, ChatGPT, Copilot, Cursor, or any context-aware AI tool.

  1. 1
    Attach the token file to your system prompt
    Paste the full contents of prism-tokens.css into the system prompt or IDE context. This gives the model the full token vocabulary before it writes a single line.
  2. 2
    Include the relevant component spec file(s)
    When building a feature involving Tooltip, attach tooltip.md. For a form, attach input.md, button.md, and label.md. Only include relevant specs — excess context dilutes focus.
  3. 3
    Write an explicit constraint in the prompt
    Start every prompt with: "Use only Prism design tokens (see attached). Do not hardcode hex values, px values, or custom class names outside the spec." Explicit constraints outperform implicit ones.
  4. 4
    Review and audit generated output
    Run the audit script against the generated code before accepting it. Flag violations back to the LLM: "This uses #1e293b — use --prism-text instead." Iterative correction tightens future outputs.
  5. 5
    Add audit to CI
    Wire the audit script into your CI pipeline. PRs that introduce hardcoded values or off-spec components fail automatically. This enforces the system without manual review.
  6. 6
    Keep spec files in sync with components
    When you update a component's API, update its spec file in the same PR. Stale spec files cause LLM drift — the model follows the spec, not the code, so outdated specs produce outdated output.

Frequently asked questions

Does this work with all LLMs?
Yes. The approach works with any model that accepts a system prompt or context window — Claude, GPT-4o, Gemini, Copilot, Cursor, etc. Larger context windows (100k+) let you include more spec files simultaneously.
What if the LLM ignores my token file?
Add an explicit instruction: "You MUST use only the tokens defined in the token file. Any hex value not listed there is a violation." Constraint language is more effective than descriptive language with LLMs.
How granular should spec files be?
Aim for one spec file per component variant group. A Tooltip spec covers all tooltip variants. A Button spec covers primary, secondary, ghost, and danger. Splitting by variant creates too many files; merging everything creates files too large to parse reliably.
Should spec files include code samples?
Yes — short, complete code samples are the single most effective addition. LLMs pattern-match on examples more reliably than they follow prose descriptions. Include one "correct" usage example and one "incorrect" example per component.
What about Figma variables — do those count as the token layer?
Figma variables are the design-side token layer. The LLM-facing token layer should be a CSS or JSON file that maps those same decisions to code-consumable variable names. Keep them in sync — a Figma variable rename should be reflected in the token file.
Who is responsible for keeping spec files current?
The component owner — the same person who updates the component source code. Spec updates should be mandatory in component PRs. Treat a stale spec file with the same severity as a broken test.

Principle

An LLM is only as good as the context you give it. Spec files make components legible. Tokens make intent portable. Audit scripts make compliance enforced. Together, they let Prism be the design system for both humans and AI.
