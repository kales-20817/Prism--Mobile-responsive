Patterns

Reusable interaction and layout patterns that solve common endpoint management workflows consistently across the UEMS product suite.

Patterns are proven solutions to recurring design problems. Unlike components (which are discrete UI elements), patterns describe how components work together to solve a user task. They encode decisions about layout, flow, interaction, and feedback that have been validated across UEMS products.

A strong pattern library reduces inconsistency, accelerates design decisions, and ensures that admins experience predictable behavior across modules.

Why patterns matter

Endpoint management products involve complex, multi-step workflows. Admins must configure policies, deploy patches, investigate threats, manage devices, and review compliance -- often under time pressure.

Without shared patterns:

  • Similar tasks feel different across modules
  • New features reinvent solved problems
  • Onboarding takes longer because nothing feels familiar
  • Cognitive load increases as admins context-switch
  • Design reviews slow down debating already-resolved decisions

Patterns create structural familiarity -- when an admin learns how policy creation works in one module, the same pattern should apply everywhere.

Pattern categories

Prism organizes patterns into six categories based on the type of problem they solve.

Layout patterns

Page structures, panel configurations, and responsive strategies for dashboards, list views, and detail panels.

Navigation patterns

Wayfinding systems including sidebar trees, breadcrumbs, tabs, and deep-link URL structures.

Data entry patterns

Forms, wizards, inline editing, and bulk actions for policy creation, device configuration, and rule setup.

Data display patterns

Tables, cards, charts, and detail views for presenting device lists, compliance scores, and threat summaries.

Feedback patterns

Notifications, toasts, progress indicators, empty states, and error recovery for async and real-time workflows.

Action patterns

Confirmation dialogs, destructive actions, batch operations, and undo/retry flows for critical endpoint tasks.

Empty states

Empty states occur when a view has no data to display. They are the most common pattern in endpoint management because admins frequently encounter unconfigured modules, empty device groups, or zero-result searches.

Anatomy

Every empty state must include these elements in order:

ElementRequiredDescription
IllustrationYesSpot illustration from the Prism library. Centered, 200-292px wide.
HeadlineYesClear statement of what's missing. Use sentence case, max 8 words.
DescriptionOptionalBrief context or next step. 1-2 lines max.
Primary CTAYesSingle action button to resolve the empty state.
Secondary linkOptionalLearn more or documentation link.

Rules

  • Always pair illustration with a clear headline
  • CTA must describe the action, not the state (e.g., "Add device" not "No devices")
  • Never show a blank white area with no guidance
  • Use consistent illustration style from the empty state library
  • Dark mode illustrations must maintain the same visual weight
Do

Illustration + "No configurations yet" + "Create configuration" button

Guide the admin toward their first action

Use a relevant spot illustration from the library

Don't

Show a blank table with "No data"

Use generic placeholder text without a CTA

Show different empty state styles per module

Form patterns

Forms are central to endpoint configuration. Admins create policies, define rules, set up alerts, and configure deployments through forms. Consistency here directly reduces setup errors.

Single-page forms

Use for simple tasks with fewer than 8 fields.

  • Group related fields with a section header
  • Place primary action at bottom-right
  • Validate inline on blur, not on submit
  • Show required indicator on optional fields, not required ones

Multi-step wizards

Use for complex configuration with logical phases (e.g., policy creation: define > target > schedule > review).

  • Show a step indicator with labels, not just numbers
  • Allow backward navigation without losing data
  • Include a review step before final submission
  • Disable "Next" until required fields are valid
  • Save draft state automatically

Inline editing

Use for quick modifications to existing records (e.g., renaming a policy, changing a threshold).

  • Click-to-edit with clear hover affordance
  • Save on blur or Enter, cancel on Escape
  • Show a subtle confirmation (checkmark) on save
  • Never use inline editing for destructive changes

Table patterns

Tables are the primary data display in UEMS. Device lists, patch inventories, compliance reports, and threat logs all use tabular layouts. Consistent table behavior is critical.

Standard table anatomy

ElementBehavior
Header rowSticky on scroll. Sortable columns show directional arrow.
Checkbox columnFirst column. Select-all in header. Indeterminate state supported.
Row hoverSubtle background highlight. Action icons appear on hover.
PaginationBottom-right. Show total count, page size selector, and page navigation.
Empty stateCentered illustration + CTA replacing table body.
LoadingSkeleton rows matching column layout. No spinner overlay.
Bulk actions barAppears above table when rows are selected. Shows count + available actions.

Column best practices

  • Name/identifier column is always first (after checkbox)
  • Status columns use semantic color badges
  • Date columns use relative time ("2 hours ago") with tooltip for absolute
  • Action column is always last, right-aligned
  • Limit visible columns to 7-8; use column picker for more

Confirmation patterns

Endpoint actions often have significant consequences: remote wipes, policy deployments, device isolation, bulk deletions. The right confirmation pattern prevents costly mistakes.

Severity levels

Low risk

Non-destructive, reversible actions.

  • No confirmation needed
  • Provide undo via toast (5s window)
  • Example: archiving a report, editing a name
Medium risk

Impactful but reversible with effort.

  • Standard confirmation dialog
  • Clear description of what will happen
  • Example: deploying a patch, changing a policy
High risk

Destructive, irreversible, or affects many endpoints.

  • Friction confirmation: type entity name to proceed
  • Red destructive button styling
  • Example: remote wipe, bulk delete, revoke certificate
Critical

Security-sensitive or compliance-affecting.

  • Multi-factor confirmation (type + re-authenticate)
  • Audit trail entry created automatically
  • Example: disable security policy, wipe all devices in group

Loading patterns

UEMS involves heavy data operations: fetching device inventories, scanning threats, generating compliance reports. Loading patterns must communicate progress without blocking productivity.

Pattern selection guide

DurationPatternExample
< 300msNo indicator neededFiltering a local list
300ms - 2sSkeleton screenLoading device table, dashboard widgets
2s - 10sProgress bar with labelPatch deployment, report generation
10s - 60sProgress with stepsBulk operations, compliance scan
> 60sBackground task + notificationFull inventory sync, OS image creation

Rules

  • Never block the entire UI for background operations
  • Skeleton screens must match the actual layout shape
  • Show estimated time remaining for tasks over 10 seconds
  • Allow cancellation for tasks over 5 seconds
  • Persist progress state across page navigations

Search and filter

Admins managing thousands of endpoints need powerful search and filter tools. These patterns must be fast, predictable, and composable.

Search bar

  • Always visible at the top of list/table views
  • Keyboard shortcut: Cmd+K or / to focus
  • Debounce 300ms before querying
  • Show result count and clear button
  • Preserve search term on navigation back

Filter bar

  • Horizontal chip-based filters below search
  • Each chip opens a dropdown for value selection
  • Applied filters show as removable tags
  • Support "Save filter" for frequently used combinations
  • "Clear all" link when any filter is active

Anti-patterns

Avoid these common mistakes when implementing patterns:

  • Inventing new confirmation flows per module instead of reusing severity levels
  • Using spinners for table loading instead of skeleton screens
  • Showing blank white space instead of proper empty states
  • Putting destructive actions in dropdown menus without confirmation
  • Creating custom wizard flows that don't support back navigation
  • Using different table layouts across modules for similar data
  • Blocking the entire page during background operations
  • Not preserving form state when navigating away and returning
  • Using inconsistent search behavior (some debounced, some instant)

Principle

In Prism, patterns encode institutional knowledge.

Every solved interaction problem becomes a reusable pattern. Admins benefit from predictable behavior, designers benefit from faster decisions, and developers benefit from consistent implementation.
