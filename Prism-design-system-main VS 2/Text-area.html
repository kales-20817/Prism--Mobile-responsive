Text Area

A multi-line text input that allows users to enter longer-form free text. It supports optional labels, placeholder text, inline tags, character counters, helper text, and validation states. Height is controlled by the maxLines prop.

Content GuidelinesRecommended

Do
  • Use Left layout (default) for standard web forms.
  • Size maxLines to match the expected content length.
  • Always show a label (even visually hidden for a11y).
  • Use helpText to give format hints (e.g. "Describe in 2-3 sentences").
  • Show counter when there is a character limit.
Don't
  • Switch to Top layout unless on a mobile/narrow breakpoint or long label.
  • Always use maxLines=2 regardless of expected input length.
  • Rely on placeholder as the only label.
  • Show an error border without an accompanying message.
  • Show a counter without a limit set in code.

When and Where to Use

Text Area is designed for multi-line free-text input where content length is variable and unpredictable.

When to use

  • Users need to enter multiple lines of free text (descriptions, comments, notes, messages).
  • Input length is variable and unpredictable — unlike a single-line text input.
  • Forms where the length or structure of the response cannot be predicted (feedback, addresses, bio).
  • A character limit needs to be communicated via the counter.

When NOT to use

  • Short, single-line input (name, email, phone) — use Text Input Field.
  • Rich/formatted text (bold, lists, links) — use Rich Text Editor.
  • Code or script input — use Script Editor.
  • Predefined answer choices — use Select Box or Radio Group.

Alignment & Placement

Consistent layout rules for label position and field sizing.

  • Left layout (default, web): Label sits to the left in a 240 px column with 8 px gap to the 320 px field. Total width: 568 px.
  • Top layout (special case): Label sits above the field. Use for responsive/mobile breakpoints or when the label exceeds ~30 characters. Width: 320 px.
  • maxLines sizing: Choose maxLines (2-5) to match expected content length. Each line adds 20 px.
  • Footer placement: Help text and counter sit in a row 4 px below the field, aligned to the field width.

Complete Do's and Don'ts

Guidelines for effective text area usage.

Do
  • Use Left layout (default): For standard web forms.
  • Use Top layout on mobile: Or when the label exceeds ~30 characters.
  • Size maxLines: Match to expected content length.
  • Always show a label: Even visually hidden for a11y.
  • Use helpText: Give format hints (e.g. "Describe in 2-3 sentences").
  • Show validationState=Error + helpText together: Always pair error borders with messages.
  • Show counter: When there is a character limit.
  • Use showClearIcon=true: For long-form fields where clearing is valuable.
  • Keep required fields marked: With required=true consistently across a form.
Don't
  • Don't default to Top layout: On desktop just because it looks familiar.
  • Don't always use maxLines=2: Regardless of expected input length.
  • Don't rely on placeholder as label: Placeholder is not a substitute for a label.
  • Don't show error border alone: Without an accompanying error message.
  • Don't show counter without a limit: Set in code.
  • Don't show clear icon: On short, single-line-height fields.
  • Don't use asterisks inconsistently: Across a form.

RTL Support

  • Set dir="rtl" on the textarea or a parent. text-align: right applies automatically.
  • Label, field content, and footer order all mirror when RTL is active.
  • Controlled via the RTL Layout variable collection in Figma.

Accessibility

Guidelines for ensuring Text Area elements are accessible across assistive technologies.

Keyboard Interaction

KeyAction
TabMoves focus into the textarea field.
Shift+TabMoves focus out of the textarea field.
Standard text-editing keysNormal text editing applies inside the field (arrow keys, selection, copy/paste, etc.).

Semantic Markup

  • Element: Use <textarea> — never a <div contenteditable>.
  • Label association: Connect <label for="id"> to <textarea id="id">, or use aria-labelledby.

ARIA Attributes & Screen Reader Guidance

  • Required: Set required / aria-required="true" on the <textarea>; the visual * alone is not sufficient.
  • Validation: Set aria-invalid="true" on the <textarea> in Error state; link error message via aria-describedby.
  • Read Only: Use readonly attribute; inform screen readers with aria-readonly="true".
  • Disabled: Use disabled attribute; disabled fields are not focusable (removed from tab order).
  • Counter: Associate counter text with the field via aria-describedby so screen readers announce remaining characters.
  • Resize: If resizable at runtime, ensure resize handle is keyboard-accessible or document the fixed height.

Focus State Design

  • Active state uses a 2 px Border/Border-Accent stroke that must be visible and not rely on colour alone.
  • The focus ring is applied to the field box (6 px border radius) and provides clear visual feedback.
