Right Pane

A fixed 48 px-wide vertical rail anchored to the right edge of an application layout, housing two stacks of icon buttons for primary utilities and supporting actions with a flexible spacer between them.

Content GuidelinesRecommended

Do
  • Keep the rail at 48 px wide — it is a design-system constant.
  • Show a tooltip on hover/focus for every icon ("Help", "Tickets", etc.).
  • Cap the top section at 5 icons and the bottom section at ~6 visible icons.
  • Swap showDarkTheme / showLightTheme to reflect the current theme.
  • Open content in an overlay, popover, or external destination — never navigate away in-place.
Don't
  • Widen the rail to accommodate labels; use tooltips instead.
  • Rely on icons alone — un-labelled icons fail accessibility.
  • Stack more than 6–7 icons per section — the rail becomes noisy and the flex spacer shrinks.
  • Show both theme icons at the same time.
  • Repurpose the named slots (Help, Tickets, Chat) for unrelated actions.
  • Treat rail icons as navigation tabs that replace the main canvas.

When and Where to Use

The Right Pane provides a persistent vertical rail for always-available utilities in an application shell.

When to use

  • As a persistent right-hand rail in an application shell that exposes always-available utilities (AI assistant, help, support, theme, announcements).
  • When primary product navigation lives in the left sidebar and secondary / contextual affordances belong on the right.
  • When you need a dense, low-chrome entry point for utilities — each icon opens a panel, popover, modal, or external destination.

When NOT to use

  • For primary navigation between major app sections → use Left Sidebar Navigation.
  • For contextual actions tied to a selected record → use an inline action bar or Split Button.
  • When the viewport is below ~768 px wide — hide the rail and move its items into a help menu in the Header Navigation.
  • For rich content panes (comments, activity, details) → use a collapsible right drawer anchored to this rail.

Complete Do's and Don'ts

Guidelines for effective Right Pane usage.

Do
  • Fixed width: Keep the rail at 48 px wide — it is a design-system constant.
  • Tooltip every icon: Show a tooltip on hover/focus for every icon ("Help", "Tickets", etc.).
  • Cap icon count: Cap the top section at 5 icons and the bottom section at ~6 visible icons.
  • Theme toggle: Swap showDarkTheme ↔ showLightTheme to reflect the current theme.
  • Extension slots: Use the Other 1–8 slots for genuinely product-specific utilities.
  • Overlay content: Open content in an overlay, popover, or external destination — never navigate away in-place.
  • Sync active state: Keep icon-button state (active) in sync with the currently open panel.
Don't
  • No labels in rail: Widen the rail to accommodate labels; use tooltips instead.
  • No unlabelled icons: Rely on icons alone — un-labelled icons fail accessibility.
  • No overcrowding: Stack more than 6–7 icons per section — the rail becomes noisy and the flex spacer shrinks.
  • No dual theme icons: Show both theme icons at the same time.
  • No slot repurposing: Repurpose the named slots (Help, Tickets, Chat) for unrelated actions.
  • No in-place nav: Treat rail icons as navigation tabs that replace the main canvas.
  • No inactive panels: Leave all icons inactive when an associated panel is open.

Accessibility

Guidelines for ensuring the Right Pane is accessible across assistive technologies and input methods.

Landmark Role

  • Wrap the rail in <nav aria-label="Utilities"> or <aside aria-label="Utilities"> so it appears as a landmark in screen-reader rotor lists.

Icon-Only Buttons

  • Every icon button must expose an aria-label matching its tooltip text (e.g. aria-label="Help"). Do not rely on the tooltip alone.
  • Tooltips appear on hover and keyboard focus, positioned to the left of the rail. Use the design system's Tooltip component.

Keyboard Interaction

ActionResult
TabMoves top-to-bottom through visible icon buttons (top section first, then bottom).
Shift+TabReverses direction.
Enter / SpaceActivates the focused icon button.
EscapeCloses any open panel triggered from the rail.

Active State & ARIA

  • When a panel triggered by the rail is open, set aria-expanded="true" (for popovers) or aria-pressed="true" (for toggles like theme) on the icon button.
  • Theme toggle: Announce the resulting state ("Switch to light theme" / "Switch to dark theme") in the aria-label, not the current state.

Focus Ring

  • Use the Icon Button focus ring (blue, 2 px) — do not remove it. The rail's white background makes the ring clearly visible.
Focus ring: 2px solid blue
Shadow-Focus-Blue token — visible against BG-Base-White

Color ContrastWCAG AA Pass

  • Icon glyphs at Icon-Tertiary must meet WCAG AA 3:1 contrast against BG-Base-White.
  • Active glyphs at Icon-Primary should meet 4.5:1.

RTL Support

  • In RTL layouts, the rail swaps to the left edge and the border moves to the right side.
  • Tooltips anchor to the right of the rail in RTL mode.

Responsive Behaviour

  • Below ~768 px viewport width, hide the rail and surface its contents in a help menu within Header Navigation.
  • Do not attempt to shrink the 48 px rail — it should be completely hidden on narrow viewports.
